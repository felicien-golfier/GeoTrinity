## Other Files
- `GeoClassChangeTrigger.h` — volume trigger for runtime class switching
- `GeoTeleporter.h` — overlap pad that teleports the touching `APlayableCharacter` to the next `AGeoTeleporter` sharing its `TeleportTag` (same-tag pads sorted by actor name, wrapping — each tag group forms its own cycle). Server does the teleport (`SetActorLocation` with `ETeleportType::TeleportPhysics`, XY of the destination pad, Z kept); the destination's `PendingArrivals` set eats arrival overlaps so they don't chain-teleport — the character stays in the set until they walk off the pad (end overlap), because begin overlap can fire more than once for a single arrival (deferred dispatch + the movement component finishing its move). **It is now a pure position move — no `ArenaTag`, no GameState call.** The camera reframes on its own because the arriving pawn overlaps the destination room's `AGeoCameraVolume`; the wipe/checkpoint entrance and the deployable push-out key off the *fighting* arena, not off where anyone teleported. This deleted the old `SetActiveArena(FindArena(ArenaTag))` write and the whole `ActiveArena`-as-"current room" concept with it. `FindNextTeleporter()` is deterministic and needs no authority, so the destination's `PendingArrivals.Add` runs on **both** client and server (only the `SetActorLocation` move stays server-gated). `TeleportTag` is EditAnywhere + BlueprintReadWrite; editor-created tags fine. In `BeginPlay`, each pad rotates (yaw only, via `SetActorRotation` on the flat 2D direction to its next pad) to face where it sends you — the mesh's arrow points at the destination; computed once at level start, not re-evaluated if teleporters move. A `UTextRenderComponent` attached to the mesh shows editor-tweakable `DisplayText` (`OnConstruction` re-syncs it, so edits are visible immediately in-editor and in-game).
- `GeoMatchStateButton.h` — overlap button; requests a configurable match-state transition (`StartMatch` / `WaitingToStart` / `WaitingPostMatch`) from `AGeoGameMode`. Server-only, mirrors `GeoClassChangeTrigger`.
- `GeoInteractableActor.h` — base for world actors that receive `FInteractableActorData`. Key fields: `Owner` (actor whose ASC drives GAS/team checks), `Instigator` (origin avatar for world position/direction — may differ from Owner). Always use `Owner` for ASC lookups; use `Instigator` for cue params and spatial calculations.
- `GeoEffectZone.h` — editor-placeable area that applies a configurable `TArray<TInstancedStruct<FEffectData>>` to actors inside its capsule. Extends `AGeoInteractableActor` (owns its own ASC), so it needs no spawner/ability — configure `Team`, `AttitudeBitmask`, `EffectDataArray`, `Radius`, `Level` in the Details panel. Self-inits GAS in `BeginPlay`; overlap delegates (`OnBeginOverlap`/`OnEndOverlap`) maintain `ActorsInZone`. `Tick()` re-applies heal/damage effects (magnitude treated as a per-second rate) to every tracked actor; other effect types are applied once on entry and removed on exit. Build heal zones (heal effect + Friendly attitude) or damage zones (damage effect + configurable `AttitudeBitmask`) purely from data.
- `GeoArena.h` — **one boss encounter, and it runs itself**; base class for every arena in a level. Holds the encounter's `BossClass`, its single `ArenaTag`, its optional `Barrier`, and its loot-shower config. **Target points are addressed by two tags, never one**: a native `TargetPoint.*` purpose (`BossSpawn`, `Entrance`, `FightLocation`, `FightCenter`, `CameraBounds`, `FallRespawn`, `FiringPoint` — all in `FGeoGameplayTags`) plus the editor-authored `Arena.*` tag naming the encounter. `GeoLib::GetTargetPoints(WorldContext, PurposeTag, ArenaTag)` requires both, so a new arena needs **only its own `Arena.*` tag** — the purposes are shared and live in code. (`CameraBounds` corner points are still resolved this way, but the *trigger* for which arena's bounds apply is now an `AGeoCameraVolume` you overlap, not the match/active-arena state; see `World/CLAUDE.md`.) One point can carry several purposes and several arena tags. The one trap: purposes and arenas multiply out, so a point carrying two purposes *and* two arena tags claims all four combinations — split it into separate points when that is not what you mean. `AGeoArena::GetArenaOfBoss(Boss)` (`Cast<AGeoArena>(Boss->GetOwner())`) is how anything holding a boss reaches its arena, and therefore its `ArenaTag`. **Exactly one boss per arena** — there is no dummy slot; a training dummy gets its own `AGeoDummyArena` instead. Spawns its boss in `BeginPlay` with `Owner = this` (so `AGeoEnemyAIController::TriggerAggro` resolves the arena off the pawn's owner) and keeps a **replicated `Boss` pointer** — clients need it for the boss health bar. `IsBoss(Actor)` is a pointer compare, not a class compare, and is `virtual` — it is the single gate on the whole match machinery, so an arena that refuses it never *starts a match*.

**The arena owns everything room-shaped — boss, barrier, fight commit, the boss health bar, and the post-victory loot shower — and it drives itself off the match state instead of being called.** The GameState holds no arena pointer at all; whether *this* arena's fight is live is the replicated **`bFighting`** flag, which `OnRep_bFighting` uses to show/hide the boss bar on every client (no GameState round-trip). Fight lifecycle:
  - `StartFight()` — called **directly by `AGeoEnemyAIController::TriggerAggro`** the moment this arena's boss is aggroed (server): sets `bFighting`, shows the boss bar, registers this arena as the GameState's respawn `CheckpointTag`, binds `Boss->OnEnemyDefeated`, closes the barrier, and arms `CommitFightTimer` from the GameState's global `CommitFightTime`. Refuses (logs and returns) while a match is already in progress — a second boss aggroed mid-fight must not steal the checkpoint, teleport the group, or leave a second `bFighting` arena behind a no-op `StartMatch`.
  - `CommitFight()` — teleports players to this arena's `TargetPoint.FightLocation` via `GeoLib::TeleportPlayersToTargetPoints`, skipping anyone already inside the `FightZoneTagName` volume.
  - `EndFight()` — clears `CommitFightTimer`, drops `bFighting`, hides the boss bar, and `ResetBoss()`es. **Reached by subscribing to `AGeoGameState::OnMatchStateChanged`** (bound in `BeginPlay`, server only): when the match leaves InProgress the arena ends its own fight; when it *enters* InProgress the arena `StopLoot()`s instead, so a fight starting anywhere shuts down every arena's shower — not just the aggroed arena's. A wipe and a victory both land here — the difference is only that a victory *also* ran the loot path first.
  `CommitFight`/`EndFight` are `virtual` — subclasses hook them to arm whatever only makes sense once the fight is really live; the commit timer dispatches through the vtable and `OnMatchStateChanged` calls `EndFight()` unqualified, so both overrides run without extra plumbing. `ResetBoss()` respawns the boss when it is gone or `ResetForNewAttempt()`s it in place.

  **Loot** is the arena's because the boss is: `OnBossDefeated` (bound to `Boss->OnEnemyDefeated` in `StartFight`) captures the loot origin, starts the looping shower (`SpawnLootBurst` every `LootSpawnInterval`, `LootPickupsPerBurst` `AGeoBuffPickup`s per burst within `LootMaxRadius`), then `RequestWaitingToStart()`s. The shower runs until the next fight starts *anywhere* — `OnMatchStateChanged` calls `StopLoot()` on every arena when the match enters InProgress (never on leaving one) — so loot keeps falling through a re-aggro, and a fight in another room still shuts a stale shower down. `GetFightingArena(WorldContext)` is the static "which arena is fighting" query (scans for the one with `bFighting`, at most one) used by the deployable push-out.

**A player's death never reaches the arena — it is `AGeoGameState`'s single policy.** `APlayableCharacter::DeathLogic` calls `AGeoGameState::NotifyPlayerDied`, and a disconnect arrives the same way because `AGeoGameMode::Logout` calls `Death()` on the leaver's pawn (Logout runs *before* `UnPossess()`, so a leaver still reads as standing otherwise). Out of a fight the player is revived on the spot; in a fight they stay down until everyone who started the fight is down (a `FightPlayers` snapshot taken in `HandleMatchHasStarted`, so late joiners neither block nor trigger a wipe), then the group respawns at `CheckpointTag` after `DeathTime`. The wipe check, the snapshot, `DeathTime`, `RevivePlayers`, `RequestWaitingToStart` all live on the GameState — see `GameClasses/CLAUDE.md`. A **victory** never goes through the wipe: the boss death runs the loot path and stands the match down directly, leaving players standing where they killed the boss. The arena's only role in death is registering *where* to respawn — `StartFight()` → `SetCheckpointTag(ArenaTag)`; a subclass could register the next room's tag instead to advance the checkpoint on victory. The one per-arena tunable left is `FightZoneTagName` (the commit exemption); `DeathTime` and the old `EntranceZoneTagName` moved to the GameState — the respawn always teleports, because there is no exempting someone from a respawn. `CommitFightTime` stays on the GameState too: it is a plain timing constant the barrier also reads, not an arena reference.
- `GeoHexArena.h` — destructible hexagonal boss platform; an `AGeoArena` whose fight is fought on breakable ground. One ISM instance per tile (~91, pointy-top axial grid, `GridRadius` rings) attached under the inherited root; tiles are **pure visuals with NoCollision** laid over a flat invisible floor: "falling" is game logic. Replicated `TArray<uint8> TileStates` is the source of truth; `ApplyTileVisuals()` diffs against `AppliedTileStates` and hides dead tiles by zero-scaling their instance (**never `RemoveInstance`** — index stability). Server calls `ApplyTileVisuals()` manually after mutating (OnReps don't fire on authority). Server API: `DestroyTiles(TConstArrayView<FIntPoint>)`, `DestroyTilesInRadius(Center, Radius)`, `ResetAllTiles()`; queries: `IsTileAlive`, `GetTileUnderLocation`, `TileToWorld`. Server `Tick` is **one rule — anything standing over a hole falls**, applied to everything inside `FallCheckRadius`: players are killed (`AGeoCharacter::Death()` + teleport of the corpse to the arena's `TargetPoint.FallRespawn` point) **except mid-dash** (`HasRootMotionSources()` — dash jumps over holes), and any `AGeoDeployableBase` is `Recall()`ed. That single rule is what makes "destroy the tile under it" the counterplay to the boss's turrets and mines, and it costs nothing extra — the tile lookup was already there. Don't add a separate registry or a tile-bound deployable base class; the fall check *is* that mechanism.
Boss abilities reach the arena with **`GetArenaOfBoss(Boss)`** (`Cast<AGeoHexArena>(Boss->GetOwner())` — arenas spawn their boss with `Owner = this`), never by class lookup. Tile queries for abilities: `IsOverAliveTile(WorldLocation)`, `GetRandomAliveTiles(Stream, Ring, Count)` (distinct tiles; `Ring < 0` or an emptied ring draws from the whole platform), `GetLastAliveTileAlongRay(Origin, Direction, MaxRange, OutTile)` (walks in half-tile steps so a clipped tile is never skipped, and stops at the far edge of the platform even when the ray outranges it), plus `GetTileRing(Tile)` and `GetGridRadius()` for ring maths. **Tick is the fall check and it is gated on the fight lifecycle**: it starts disabled (`bStartWithTickEnabled = false`), `CommitFight()` enables it, `EndFight()` disables it and resets the tiles. That gate is the whole point — before commit nobody has been teleported onto the platform, so players standing in the void around it (or still walking in) must not be killed. Don't add a separate "is the fight live" flag; the tick enable *is* that state.
- `GeoDummyArena.h` — an `AGeoArena` for a training dummy: it spawns its enemy at level start and then just stands there. Header-only, and the whole class is now one one-liner: `IsBoss() → false`. `EntranceArena` in `DraftMap` is one: the hub's training dummy, and the hub itself (`Arena.Entrance`). `IsBoss` is genuinely sufficient to stay out of the match: `AGeoEnemyAIController::TriggerAggro` is the **only** place that calls `StartMatch()`, and it asks `IsBoss` first — so refusing there means no match, no `StartFight` on this arena, so `bFighting` never goes true and every downstream effect (boss health bar, `OnEnemyDefeated` → loot binding, barrier, fight commit) never fires. Like any arena it *does* subscribe to `OnMatchStateChanged` in `BeginPlay`, but with `bFighting` stuck false its `EndFight` is never reached — the subscription is inert. There is no `RespawnPlayer` override any more: because the dummy never runs a match, a death here is always "out of a fight", which `AGeoGameState::NotifyPlayerDied` already handles by reviving on the spot — the instant rez that used to need a per-arena override is now the universal out-of-fight policy. Don't add no-op `StartFight`/`CommitFight`/`EndFight` overrides "for safety": they are unreachable on a dummy arena and would be dead code. Only `BossClass` and `ArenaTag` are meaningful on it — `ArenaTag` still matters because its `TargetPoint.BossSpawn` point is resolved through it, and the hub's entrance points hang off the same tag; `Barrier` is never read because the arena never runs a fight. The hub is an arena like any other, but the game no longer tracks a "current arena" at all — where you are is answered by the camera volume you stand in and, during a fight, by whichever arena has `bFighting` set.
- `GeoArenaBarrier.h` — replicated barrier that blocks the arena entrance; opens/closes via `SetClosed(bool)` (server-only); animated actors lerp full `FTransform` between `FightOnTransform` and `FightOffTransform` — **closing over `CommitFightTime`, opening over `DeathTime`** (both read from `AGeoGameState`): on a wipe `AGeoGameState` broadcasts `OnWipe(DeathTime)` the moment everyone is down; the fighting arena's handler (`AGeoArena::OnWipe`, gated on `bFighting`) cancels the pending commit and `SetClosed(false)`s, so the barrier spends the death window opening and finishes right as the group respawns. A victory opens it at `EndFight` over the same `DeathTime`. Editor helpers: `CaptureFightOn/OffTransforms`, `SetToFightOn/OffTransforms`. The lerp is driven by the actor's own `Tick`, enabled from `OnRep_bIsClosed` (the single state-change seam, called manually by `SetClosed` on the server since OnReps don't fire on the authority) and disabled once `LerpAlpha` settles at its target. Drive it *only* that way — the previous `SetTimerForNextTick` chain had a tautological stop condition (`LerpAlpha` is clamped to `[0,1]`, so `bIsClosed && LerpAlpha <= 1 || !bIsClosed && LerpAlpha >= 0` is always true), so no chain ever terminated and every redundant entry point spawned a parallel one that each advanced the alpha once per frame. The barrier closed N× too fast, N = live chain count, accumulating with every open/close. `OnBarrierStateChanged` is a pure `BlueprintImplementableEvent` for VFX only — never hang lerp driving off it, or a BP override that forgets the parent call freezes the barrier. `LerpAlpha` is protected so subclasses can render the open/close progress their own way.
- `GeoHexBarrier.h` — an `AGeoArenaBarrier` whose blocking element is an alley floor of hex tiles (same pointy-top layout, tile scale, and zero-scale-hide trick as `AGeoHexArena`, one ISM instance per tile, NoCollision). Instead of moving actors, it hides tiles one by one as the inherited `LerpAlpha` climbs (close) and restores them as it falls (open) — `Tick` calls `Super::Tick` then `ApplyTileVisuals`, which diffs the hidden count (`RoundToInt(LerpAlpha * NumTiles)`) against `AppliedHiddenCount` and touches only the changed index range. No replication of its own: the lerp is already driven per-machine off the replicated `bIsClosed`, so every client vanishes the same tiles in the same order for free. Tiles vanish one by one in ISM index order; `bVanishAlongColumns` picks the sweep — column by column along local +X (true) or row by row along local +Y (false) — by changing the `Index`→(Column, Row) mapping in `GetTileTransform(Index)`, which both `OnConstruction` and `ApplyTileVisuals` share. Rotate the actor for other directions. `NumColumns` / `NumRows` / `TileSize` rebuild the floor in `OnConstruction`, so resizing in-editor regenerates instantly; match `TileSize` to the arena's. The ISM is the root component (the base class creates no root). Purely visual like the arena tiles — if players must fall in the gap, the alley has to sit inside the arena's `FallCheckRadius`.
